# Task 2.1: Mod Architecture Design

**Phase**: 2 - Companion Mod Development
**Status**: Complete

## Summary

Design minimal, portable mod architecture for multi-version support.

## Why This Matters

A clean abstraction layer makes porting between 1.7.10/1.12.2/1.19+ manageable.

## Inputs/Outputs

- **Input**: Version differences research
- **Output**: Architecture design document

## Technical Notes

**Design Philosophy**: Keep the mod as minimal and portable as possible. Abstract version-specific code behind interfaces.

**Target Versions**:
- 1.7.10 (GTNH - uses NEI)
- 1.12.2 (Common GT version - uses JEI)
- 1.19+ (Modern GT - uses EMI/JEI)

**Core abstraction layer (version-agnostic)**:
- `RecipeProvider` interface
- `RecipeData` model (JSON-serializable)
- `HttpExporter` utility
- Command handler interface

**Version-specific implementations**:
- 1.7.10: NEI recipe handler integration
- 1.12.2: JEI recipe registry integration
- 1.19+: EMI/JEI recipe registry integration

**Minimal dependencies**: just HTTP client + JSON
**Single command**: `/recipeflow sync`
**Config**: server URL, auth token, modpack ID

## Files to Touch

- `mod/core/` (shared abstractions)
- `mod/versions/1.7.10/`
- `mod/versions/1.12.2/`
- `mod/versions/1.19+/`

## Acceptance Criteria

- [x] Clean interface defined for recipe extraction
- [x] Version-specific code isolated
- [x] Common code maximized
